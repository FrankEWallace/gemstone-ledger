---
title: Database Migrations
description: How schema changes are managed and applied to the Supabase project.
---

All schema lives in `supabase/migrations/`, numbered `001`–`025`. The files are the source of truth for *what* the schema is, but they are **not** applied with `supabase db push`.

<Warning>
  Do not run `supabase db push` against the production project. The remote migration history records these files under 14-digit timestamp versions while the local files use short numeric prefixes, so the CLI cannot reconcile the two and will attempt to re-apply `001_initial_schema.sql` onto the live database. (A CI job that did this was removed for exactly this reason.)
</Warning>

## Applying a new migration

New migrations are applied directly to the remote project, then the same SQL is committed to `supabase/migrations/` as a mirror:

1. Write the migration SQL as an idempotent script (guards like `if not exists`, `drop ... if exists`, `create or replace`).
2. Apply it via the Supabase MCP tooling or the dashboard **SQL Editor**.
3. Commit the file with the next number so the repo history matches production.

Before writing a migration that touches existing objects, verify the actual production state first (columns, function grants, policies) — the deployed schema has diverged from the numbered files in the past.

## Migration history

| Range | What it covers |
|---|---|
| 001–005 | Core schema (orgs, sites, users, inventory, transactions), RLS fixes, audit |
| 006–010 | Team/equipment/safety/documents, audit log, alert rules, supply chain, campaigns, customers |
| 011–016 | Module config, signup RPC, write-offs, maintenance logs, incident resolution, documents storage bucket |
| 017–021 | Org-level roles (owner/admin/member), notification prefs, owner-on-signup trigger, currency reconciliation, org users RPC |
| 022–023 | Scheduled edge functions (pg_cron) and Vault-backed cron secret lockdown |
| 024–025 | Security-advisor hardening: pinned search_path, dropped over-permissive policies, closed a role self-escalation hole, locked down the signup RPC surface |

## Fresh local stack

For a brand-new local database the numbered files do apply cleanly in order:

```bash
supabase start
supabase db reset   # applies supabase/migrations/ from scratch
```
